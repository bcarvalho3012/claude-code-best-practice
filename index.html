Level 1: The Essentials

The five building blocks every Claude Code user should know

Memory (CLAUDE.md)

Think of it as: A sticky note your AI reads every morning

What It Is

CLAUDE.md is a plain-text file you place in your project root. Claude reads it automatically at the start of every session, so anything you write there becomes persistent instructions. Think of it as your project's house rules — coding standards, stack details, and workflow preferences that Claude should always remember.

How to Use It

  1. Create a file called CLAUDE.md in your project root
  2. Write plain-English rules — for example, "This project uses Python 3.11" or "Always write tests for new functions"
  3. Keep it under 200 lines so Claude reliably follows every instruction
  4. That's it — Claude reads it automatically at the start of every session

Example

# CLAUDE.md

## Project Overview
This is a FastAPI backend using Python 3.11 and PostgreSQL.

## Rules
- Always use type hints in function signatures
- Write pytest tests for every new endpoint
- Use snake_case for variables and functions
- Never commit .env files
- Keep functions under 30 lines
Advanced Details

Memory Hierarchy

Claude reads CLAUDE.md files at multiple levels, from broadest to most specific:

LocationScopeExample Use
~/CLAUDE.mdGlobal (all projects)Personal coding style preferences
./CLAUDE.mdProject rootProject-wide stack and conventions
./src/CLAUDE.mdSubfolderModule-specific rules
.claude/rules/*.mdGlob-scoped rulesRules that apply only to matching files

The /memory Command

Use the built-in /memory command to quickly edit your CLAUDE.md without leaving the Claude Code session. Claude can also write to its own memory files to remember things across sessions.

Deep dive: claude-memory.md

Commands

Think of it as: Speed dial buttons for your AI

What It Is

Commands are pre-written instructions you trigger by typing / followed by a name. Instead of typing "review this PR and check for security issues" every time, you create a command once and type /review forever after. They live as markdown files in your project's .claude/commands/ folder.

How to Use It

  1. Create a .claude/commands/ folder in your project
  2. Add a markdown file — the filename becomes the command name (e.g., review.md becomes /review)
  3. Write instructions inside the file, with optional YAML frontmatter for settings
  4. Type /review in Claude Code to run it

Example

# .claude/commands/review.md
---
description: Review code for bugs and security issues
model: sonnet
allowed-tools: Read, Grep, Glob
---

Review the current git diff for:
1. Security vulnerabilities (SQL injection, XSS, etc.)
2. Logic errors and edge cases
3. Missing error handling

Provide a summary with severity ratings.

Skills

Think of it as: Instruction manuals Claude can pick up and follow

What It Is

Skills teach Claude how to do specific things — generate SVGs, write migration scripts, create API endpoints in your project's style. They live in .claude/skills/ folders as SKILL.md files. Some skills are auto-discovered by Claude when relevant; others you trigger explicitly with /skill-name.

How to Use It

  1. Create a .claude/skills/your-skill-name/ folder
  2. Add a SKILL.md file inside that folder
  3. Write instructions and examples in the file, with YAML frontmatter for configuration
  4. Claude discovers it automatically based on the description field, or you trigger it with /your-skill-name

Example

# .claude/skills/api-endpoint/SKILL.md
---
name: api-endpoint
description: "Create a new REST API endpoint following project conventions"
allowed-tools: Read, Write, Grep, Glob, Bash
---

## Instructions

When creating a new API endpoint:
1. Create the route handler in `src/routes/`
2. Add request/response schemas in `src/schemas/`
3. Write tests in `tests/routes/`
4. Register the route in `src/app.py`

Follow the patterns in existing endpoints for consistency.

Sub-Agents

Think of it as: Delegating to a coworker who works independently and reports back

What It Is

Sub-agents are independent copies of Claude that handle specific subtasks. Each one gets its own context window, can use a different AI model, and has restricted tool access. You define them as markdown files in .claude/agents/, and Claude uses the Agent tool to spin them up when needed.

How to Use It

  1. Create a .claude/agents/ folder in your project
  2. Add a markdown file for your agent (e.g., test-runner.md)
  3. Define its capabilities in YAML frontmatter — tools it can use, which model, and preloaded skills
  4. Claude invokes it automatically via the Agent tool, or auto-invokes it if the description contains "PROACTIVELY"

Example

# .claude/agents/test-runner.md
---
name: test-runner
description: "Runs tests and reports results. Invoke PROACTIVELY when code changes are made."
model: haiku
tools: Bash, Read, Glob
maxTurns: 10
---

## Instructions

You are a test runner agent. When invoked:
1. Identify which test files are relevant to the changed code
2. Run the tests using `pytest`
3. Report results: passed, failed, and any error messages
4. Suggest fixes for failing tests

Hooks

Think of it as: A doorbell that rings when someone arrives

What It Is

Hooks are scripts that run automatically when specific events happen in Claude Code. Want to play a sound when Claude finishes a task? Run a linter after every file edit? Send a Slack notification when a long task completes? Hooks make all of that possible by connecting your scripts to Claude's lifecycle events.

How to Use It

  1. Open .claude/settings.json in your project
  2. Add hook entries under specific event names (e.g., PreToolUse, PostToolUse, Stop)
  3. Point each hook to a script file or an inline command
  4. They run automatically at the right moment — no manual triggering needed

Example

// .claude/settings.json
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "echo 'Claude finished!' && afplay /path/to/done.mp3"
      }
    ],
    "PostToolUse": [
      {
        "type": "command",
        "command": "python .claude/hooks/scripts/hooks.py",
        "matcher": "Write"
      }
    ]
  }
}
Advanced Details

Available Hook Events

Claude Code supports 19 hook events. The most commonly used are:

EventWhen It Fires
PreToolUseBefore Claude uses a tool (can block or modify)
PostToolUseAfter a tool finishes (great for linting, formatting)
StopWhen Claude finishes its response
NotificationWhen Claude sends a notification
SessionStartWhen a new Claude Code session begins
SessionEndWhen a session ends
UserPromptSubmitWhen you submit a prompt
TaskCompletedWhen a task finishes

Disabling Hooks

Set "disableAllHooks": true in .claude/settings.local.json to turn off all hooks for your local environment. You can also disable individual hooks in the configuration files.

Level 2: Structured Workflows

Combining the essentials into powerful, repeatable patterns

Workflows

Think of it as: An orchestra conductor coordinating musicians

What It Is

A workflow is the Command → Agent → Skill architecture pattern. A command handles user interaction, an agent fetches and processes data, and a skill produces the final output. Each component has a single responsibility, making the system modular and reusable.

How to Use It

  1. Create a command as the entry point — it handles user input and orchestrates the flow
  2. Create an agent to do the heavy lifting — fetching data, processing, making decisions
  3. Create a skill for the output — generating files, rendering visualizations, writing reports
  4. Wire them together: the command asks the user for input, invokes the agent, then invokes the skill with the results

Example

┌─────────────────────────────────────────────────────┐
│  /weather-orchestrator (Command)                    │
│  Entry point — asks: "Celsius or Fahrenheit?"       │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │  weather-agent (Agent)                        │  │
│  │  Fetches temperature from Open-Meteo API      │  │
│  │  └── weather-fetcher (Preloaded Skill)        │  │
│  └───────────────────────────────────────────────┘  │
│                         │                           │
│                         ▼                           │
│  ┌───────────────────────────────────────────────┐  │
│  │  weather-svg-creator (Skill)                  │  │
│  │  Creates a visual weather card as SVG         │  │
│  │  Writes: weather.svg + output.md              │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
Advanced Details

Two Skill Patterns

PatternHow It WorksWhen to Use
Agent Skills (preloaded) Listed in an agent's skills: frontmatter field; instructions are loaded into the agent's context automatically When the agent always needs specific knowledge to do its job
Skills (invoked) Triggered via the Skill tool during execution, either by the command or agent When skills are optional or shared across multiple workflows

Design Principle

Each component should have a single responsibility. Commands handle interaction, agents handle processing, skills handle output. This separation makes components reusable — the same agent can be used by different commands, and the same skill can be invoked by different agents.

Settings

Think of it as: House rules that everyone follows

What It Is

Settings files control Claude's behavior — what it can do automatically, what needs your permission, which AI model to use, and how hooks are configured. Settings follow a hierarchy: team settings are shared via git, while personal settings override them locally and stay git-ignored.

How to Use It

  1. Open or create .claude/settings.json for team-shared settings
  2. Add permission rules using allow, ask, and deny arrays
  3. Set model preferences and other behavioral options
  4. Create .claude/settings.local.json for personal overrides (automatically git-ignored)

Example

// .claude/settings.json (shared with team)
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(npm test)",
      "Bash(npm run lint)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)"
    ]
  },
  "model": "sonnet"
}
Advanced Details

Settings Hierarchy (Highest to Lowest Priority)

PrioritySourcePurpose
1 (highest)Managed settings (MDM / Registry)Organization-enforced, cannot be overridden
2Command line argumentsSingle-session overrides
3.claude/settings.local.jsonPersonal project settings (git-ignored)
4.claude/settings.jsonTeam-shared project settings
5 (lowest)~/.claude/settings.jsonGlobal personal defaults

Scale

Claude Code supports 55+ settings keys and over 140 environment variables for fine-grained control. Most teams only need permissions, model selection, and hook configuration to get started.

MCP Servers

Think of it as: Plugging in accessories to your computer

What It Is

MCP (Model Context Protocol) is a standard way to connect Claude to external tools and services — web browsers, documentation APIs, databases, and more. You configure MCP servers in a .mcp.json file, and Claude automatically connects to them on startup, gaining new capabilities without any custom code.

How to Use It

  1. Create or edit .mcp.json in your project root
  2. Add server configurations with the server name, command, and arguments
  3. Claude automatically connects to the servers on startup
  4. Use /mcp to check connection status and manage servers

Example

// .mcp.json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@anthropic/mcp-playwright@latest"],
      "env": {}
    },
    "context7": {
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp@latest"]
    }
  }
}
Advanced Details

Popular MCP Servers

ServerWhat It Does
PlaywrightBrowser automation — Claude can navigate pages, click buttons, take screenshots
Context7Live documentation lookup — Claude can query up-to-date docs for any library
DeepWikiRepository documentation — Claude can read generated wiki pages for GitHub repos

MCP in Settings

MCP servers can also be configured in .claude/settings.json under the mcpServers key, and individual sub-agents can declare their own MCP servers in their frontmatter. Use .mcp.json for project-wide servers and agent-level config for specialized tools.

Checkpointing

Think of it as: Video game save points

What It Is

Claude automatically saves snapshots of your files as it works, creating restore points you can rewind to if something goes wrong. If Claude makes a change you don't like, you can undo everything back to an earlier checkpoint — no manual git stashing or backup copying required.

How to Use It

  1. Claude creates checkpoints automatically as it edits files — you don't need to configure anything
  2. If something goes wrong, press Esc to interrupt Claude
  3. Press Esc Esc (twice quickly) to open the rewind interface
  4. Select a checkpoint from the list to restore your files to that point in time

Example

Keyboard Shortcuts:
─────────────────────────────────────────────
  Esc           →  Interrupt Claude mid-response
  Esc Esc       →  Open the rewind/checkpoint UI
─────────────────────────────────────────────

Command:
─────────────────────────────────────────────
  /rewind       →  Rewind to a previous checkpoint
─────────────────────────────────────────────
Advanced Details

How Checkpoints Work

Checkpoints are created before each file modification. They capture the state of all files that Claude is about to change. When you rewind, Claude restores those files to their exact state at the selected checkpoint. This works alongside git — checkpoints are session-level restore points, while git commits are permanent history.

The /rewind Command

You can also type /rewind at any time to access the checkpoint interface without using keyboard shortcuts. This is especially useful if you realize several steps later that an earlier change was wrong.

Built-in feature — no additional configuration needed

Think of it as: Car dashboard dials you set before driving — you adjust mirrors, AC, and seat position before you start the engine.

What It Is

Command-line options you add when starting Claude Code to customize behavior for that session. Things like choosing a model, enabling plan mode, or running a single command without entering the interactive session.

Flags let you tailor each session without changing your permanent configuration. They override your settings for that run only.

How to Use It

  1. Open your terminal
  2. Type claude followed by any flags you want
  3. Common flags adjust model, mode, verbosity, and more
  4. The flags only apply to that session — next time you start fresh

Example

# Use a specific model claude --model opus # Start in plan mode (Claude explains before acting) claude --plan # Ask a one-off question without entering interactive mode claude --print "What does this regex do? /^[a-z]+$/i" # See detailed logging for debugging claude --verbose # Start with a specific agent claude --agent code-reviewer # Restrict which tools Claude can use claude --allowedTools "Read,Glob,Grep"
Advanced Details

Run claude --help to see the full list of available flags. Here are the most useful ones:

  • --model — Choose which AI model to use (haiku, sonnet, opus)
  • --plan — Start in plan mode so Claude explains its approach before making changes
  • --print "question" — Get a one-shot answer without entering the interactive session
  • --verbose — Show detailed logs, useful for debugging hooks or agents
  • --agent <name> — Start with a specific agent loaded
  • --allowedTools — Restrict Claude to only certain tools for the session

Flags take precedence over settings files but are lower priority than managed (organization-enforced) settings.

What It Is

Downloadable packages that add new skills, agents, hooks, or tools to your Claude Code setup. They can be created by the community or your organization and are installed via the /plugin command from plugin marketplaces.

Instead of writing every skill or agent yourself, you can install pre-built plugins that others have already perfected.

How to Use It

  1. Run /plugin inside Claude Code to browse available plugins
  2. Select a plugin from the marketplace and install it
  3. The plugin's skills, agents, and hooks become immediately available
  4. Use /reload-plugins to refresh plugins after updates

Example

// In .claude/settings.json — plugin configuration { "enabledPlugins": [ "code-review", "security-scanner", "doc-generator" ], "extraKnownMarketplaces": [ "https://my-company.com/claude-plugins" ] }
Advanced Details

Plugins are distributed through marketplaces — curated registries of verified plugins. You can use the default public marketplace or add your organization's private marketplace via extraKnownMarketplaces in settings.

  • enabledPlugins — List of installed and active plugins
  • extraKnownMarketplaces — Additional plugin registries beyond the default
  • Plugins can include any combination of skills, agents, hooks, commands, and rules
  • Use /reload-plugins after updating plugin source files

Organizations can create internal plugins to standardize workflows across teams.

Think of it as: Dashboard gauges showing your car's status — speed, fuel level, engine temperature — all visible at a glance without opening the hood.

What It Is

A customizable status bar at the bottom of the Claude Code terminal that shows real-time information like your current git branch, context window usage, or any custom data you want. It's configured via a shell script that outputs the data to display.

Instead of running commands to check status, the information is always visible as you work.

How to Use It

  1. Create a shell script that outputs the status info you want to see
  2. Add a statusLine configuration to your settings.json
  3. The status bar appears at the bottom of Claude Code and updates in real-time

Example

// In .claude/settings.json { "statusLine": { "command": "echo \"Branch: $(git branch --show-current) | Context: ${context_window.used_percentage}%\"", "interval": 5 } } // Output shown at bottom of terminal: // Branch: feature/login | Context: 34%
Advanced Details

The status line script receives several input fields as environment variables that you can use:

  • context_window.used_percentage — How much of Claude's context window is used
  • workspace.added_dirs — Directories added to the current workspace
  • Any shell command output — git branch, disk usage, running processes, etc.

The interval setting controls how often the status refreshes (in seconds). Keep it reasonable to avoid performance impact.

Deep dive: claude-statusline.md

What It Is

Three levels of AI-assisted development that describe how much structure and context you provide to Claude. Understanding these levels helps you know where you are and where you can go.

Most beginners start at Level 1 (Vibe Coding) and gradually progress as they learn to give Claude better context and structure.

The Three Levels

Level What You Do What Claude Does
Vibe Coding Type “add a login page” Makes its best guess, often inconsistent with your project
Context Engineering Provide CLAUDE.md, commands, skills Follows your project’s patterns, conventions, and architecture
Agentic Engineering Set up agents, hooks, orchestration Works autonomously with guardrails, multi-agent coordination

Example

# Vibe Coding (Level 1): "Add user authentication" # Claude guesses your stack, patterns, file locations # Context Engineering (Level 2): # You have CLAUDE.md explaining your auth patterns, # a /add-auth command, and skills for your ORM "Add user authentication" # Claude follows YOUR patterns exactly # Agentic Engineering (Level 3): # You have a security-agent, test-agent, and review-agent # that coordinate via hooks and shared context "Add user authentication" # Multiple agents build, test, and review the feature
Advanced Details

These terms were coined by the Claude Code community to describe the progression of sophistication in AI-assisted development:

  • Vibe Coding — Quick and easy, but results are unpredictable. Fine for prototyping and throwaway code.
  • Context Engineering — The sweet spot for most teams. Invest time in CLAUDE.md and commands to get consistent, high-quality output.
  • Agentic Engineering — For complex projects and teams. Requires upfront investment in agent architecture but unlocks autonomous workflows.

You don't have to pick just one level. Most developers use all three depending on the task at hand.

Think of it as: Whispering a side question to a coworker without interrupting their work — they answer you, then go right back to what they were doing.

What It Is

A built-in command that lets you ask Claude a quick question without it becoming part of the conversation memory. It's perfect for checking something without derailing the current task Claude is working on.

The answer appears separately and doesn't influence Claude's ongoing work or context.

How to Use It

  1. While Claude is working on a task (or between prompts)
  2. Type /btw followed by your question
  3. Claude answers your question in a separate context
  4. The main conversation continues unaffected

Example

# Claude is in the middle of refactoring your auth module... /btw how do I format dates in Python using strftime? # Claude answers: Use datetime.strftime("%Y-%m-%d") for ISO format... # Then continues the refactoring without the date question in its memory /btw what's the keyboard shortcut for splitting a terminal in VS Code? # Claude answers separately, main task stays on track
Advanced Details

The /btw command is one of Claude Code's built-in commands. It's especially useful when:

  • You need a quick syntax reminder but don't want to pollute the conversation
  • You want to ask about something unrelated to the current task
  • You're deep in a complex refactoring and need a side answer

Since the question and answer don't enter the main context, they won't consume your context window budget or affect Claude's understanding of the current task.

Level 4: Pro Features

Expert-level capabilities for maximum productivity and automation

Think of it as: A recurring alarm on your phone — set it once, and it goes off at the same time every day until you turn it off.

What It Is

Have Claude repeat a task on a schedule — like monitoring a deployment, running tests periodically, or checking for changes in a remote service. Uses the /loop skill to run commands at regular intervals.

It runs for up to 3 days, making it great for long-running monitoring tasks without needing a separate cron job.

How to Use It

  1. Type /loop followed by an interval and your command
  2. Claude runs the command immediately, then repeats at the specified interval
  3. Results are reported each time the command runs
  4. The loop continues for up to 3 days or until you stop it

Example

# Check deployment status every 5 minutes /loop 5m check if the deploy on staging succeeded # Run tests every 10 minutes during a refactor /loop 10m run the test suite and report any failures # Monitor a log file for errors every minute /loop 1m check /var/log/app.log for new errors # Check API health every 30 seconds /loop 30s curl https://api.myapp.com/health and report status
Advanced Details

The /loop skill is a built-in capability that supports various interval formats:

  • 30s — every 30 seconds
  • 5m — every 5 minutes
  • 1h — every hour
  • Maximum runtime is 3 days before the loop automatically stops
  • You can combine /loop with any command or skill

What It Is

Two built-in skills that save time on common tasks. /simplify reviews your recently changed code for quality issues and duplication, then suggests improvements. /batch runs a command across multiple files in bulk.

What It Is

Temporary copies of your repository that let agents work on different branches without interfering with each other. When an agent has isolation: "worktree", it gets its own complete copy of the codebase to work in.

Changes stay isolated until you explicitly merge them, so there's no risk of file conflicts between parallel agents.

How to Use It

  1. Set isolation: "worktree" in your agent's frontmatter
  2. When the agent runs, it automatically gets its own copy of the repo
  3. The agent works freely without affecting your main working directory
  4. If the agent makes changes, they stay on a separate branch for you to review and merge
  5. If no changes were made, the worktree is auto-cleaned

Example

# In .claude/agents/refactor-agent.md --- name: refactor-agent description: Refactors code modules for better structure isolation: "worktree" model: sonnet --- Refactor the specified module for clarity and reduced duplication. Work in your isolated worktree — your changes won't affect the main branch until explicitly merged. # When this agent runs, it: # 1. Creates a temporary git worktree (separate directory) # 2. Works on a new branch in that directory # 3. Makes all its changes there # 4. Reports back — you decide whether to merge
Advanced Details

Git worktrees are a native Git feature that Claude Code leverages for agent isolation:

  • Each worktree is a full checkout of the repo at a separate path
  • The agent gets its own branch automatically
  • Multiple agents with isolation: "worktree" can run in parallel without conflicts
  • Worktrees are automatically cleaned up if the agent makes no changes
  • Combine with teammateMode: "tmux" for parallel visual workflow

This is the recommended approach for agent teams that need to edit the same files — each agent works in isolation, and you merge the results.

What It Is

A meta-pattern where you use Claude Code to improve Claude Code's own configuration. Ask Claude to review and refine your CLAUDE.md, suggest better commands, tune agent definitions, and optimize skills — all with Claude's help.

Named humorously after the recursive, self-referential nature of using the tool to improve the tool. It's how power users continuously level up their setup.

How to Use It

  1. Ask Claude to review your CLAUDE.md for gaps or improvements
  2. Ask it to suggest new commands or skills based on your workflow
  3. Use /init to bootstrap a project with sensible defaults
  4. Iterate: use Claude to improve how Claude works for you
  5. Repeat regularly as your project evolves

Example

# Ask Claude to review its own instructions: "Review my CLAUDE.md and suggest improvements. What patterns am I missing? What context would help you do a better job on this project?" # Claude might respond: # "Your CLAUDE.md is missing: # 1. Database schema context — I keep guessing column names # 2. Error handling conventions — you use Result types but # I default to try/catch # 3. A /deploy command would save you from repeating # the 5-step deploy process every time" # Then ask Claude to implement its own suggestions: "Go ahead and update the CLAUDE.md with those improvements and create that /deploy command" # The loop: Claude improves its own context → works better → # you spot new improvements → Claude refines further
Advanced Details

The Ralph Wiggum Loop is most effective when practiced regularly:

  • After onboarding — Run /init then immediately ask Claude what's missing
  • After repeated friction — If you keep correcting Claude on the same thing, ask it to add that knowledge to CLAUDE.md
  • After new features — When your project grows, ask Claude to update its own context
  • Weekly reviews — Periodically ask "What would make you more effective on this project?"

This is the core practice of Context Engineering — systematically improving the context you give Claude so it produces better results over time.

Weather Walkthrough

Trace a real workflow from start to finish

Let's trace a real workflow from start to finish. The weather orchestration example shows how a Command, an Agent, and two Skills coordinate to fetch live weather data and render a visual card — each component doing exactly one thing.

1 User types /weather-orchestrator

The command file is the entry point. It asks the user a question, then delegates work to an agent and a skill.

# File: .claude/commands/weather-orchestrator.md --- description: Fetch weather data for Dubai and create an SVG weather card model: haiku --- Step 1: Ask user — Celsius or Fahrenheit? Step 2: Invoke weather-agent via Agent tool Step 3: Invoke weather-svg-creator via Skill tool

2 Agent fetches data

The command invokes a sub-agent via the Agent tool. This agent has a skill preloaded — it already knows how to fetch weather data.

# File: .claude/agents/weather-agent.md --- name: weather-agent model: sonnet skills: - weather-fetcher --- Follow the weather-fetcher skill to fetch temperature. Return the value to the caller.

3 Agent uses its preloaded skill

The agent doesn't need to figure out the API — its preloaded skill has exact instructions.

# File: .claude/skills/weather-fetcher/SKILL.md --- name: weather-fetcher user-invocable: false --- Fetch from: api.open-meteo.com Dubai coordinates: 25.2048, 55.2708 Return: temperature + unit

4 SVG creator makes the output

Back in the command, the orchestrator invokes the SVG skill via the Skill tool. This skill creates the final deliverables.

# File: .claude/skills/weather-svg-creator/SKILL.md --- name: weather-svg-creator --- Create SVG weather card → weather.svg Write summary → output.md

Complete Flow

User types: /weather-orchestrator | +-- Step 1: "Celsius or Fahrenheit?" → User picks | +-- Step 2: Agent(weather-agent) | +-- Preloaded skill: weather-fetcher | +-- Fetches from Open-Meteo → 32°C | +-- Step 3: Skill(weather-svg-creator) | +-- Creates: weather.svg | +-- Writes: output.md | +-- Done! Shows summary to user

Key Takeaway

Command orchestrates, Agent fetches, Skill renders — each does one thing well. This separation keeps every component simple, testable, and reusable.

File Structure Atlas

Where everything lives inside your project

your-project/ +-- CLAUDE.md # Project memory +-- .mcp.json # MCP server config +-- .claude/ +-- settings.json # Team settings +-- settings.local.json # Personal settings (git-ignored) +-- commands/ # Slash commands | +-- my-command.md +-- agents/ # Sub-agents | +-- my-agent.md +-- skills/ # Skills | +-- my-skill/ | +-- SKILL.md +-- rules/ # Context-specific rules | +-- my-rule.md +-- hooks/ # Hook scripts +-- my-hook.sh
CLAUDE.md — Project Memory

A markdown file at the root of your project that Claude reads at the start of every session. Use it for persistent instructions: tech stack, coding conventions, things to avoid. Keep it under 200 lines for best results.

.mcp.json — MCP Server Configuration

Declares which external tools (browsers, databases, APIs) Claude can connect to via the Model Context Protocol. Checked into version control so the whole team shares the same tool setup.

commands/ — Slash Commands

Each .md file becomes a /command-name shortcut. The filename is the command name. Use YAML frontmatter at the top for settings like model, description, and allowed-tools. The body contains the instructions Claude follows when the command is triggered.

agents/ — Sub-Agents

Define independent copies of Claude that handle specific subtasks. Each agent can have its own model, permissions, skills, and tool allowlist. Invoked via the Agent tool from commands or other agents. They work in isolation and return results to the caller.

skills/ — Skills

Instruction manuals for specific tasks. Each skill lives in its own subdirectory with a SKILL.md file. Skills can be preloaded into agents (agent skills) or invoked on demand via the Skill tool. Set user-invocable: false to hide from the / menu.

rules/ — Context-Specific Rules

Instruction files that Claude follows automatically when working with matching files or directories. Use a # Glob: pattern header to specify which files the rule applies to. Like house rules that activate in certain rooms.

hooks/ — Hook Scripts

Scripts triggered by lifecycle events: tool use, notifications, session start/end, git commits, and more. Configured in settings.json. Use them for automation like playing sounds, logging, or validation gates.

settings.json vs settings.local.json

settings.json is shared with the team (checked into git). settings.local.json is for personal overrides (git-ignored). The local file takes priority. Use this split to keep team-wide defaults while allowing individual preferences.

Glossary

Plain-English definitions for every term

Agent

See Sub-Agent.

Agent Teams

Multiple copies of Claude working on different parts of the same project at the same time, like a team splitting up tasks.

Agentic Loop

The repeating cycle Claude follows: look at the situation, decide what to do, take action, check the result, and repeat until done.

Checkpointing

Claude automatically saves snapshots of your files as it works, so you can undo changes if something goes wrong. Press Esc Esc to rewind.

CLI

Command Line Interface — the text-based terminal window where you type commands instead of clicking buttons.

CLAUDE.md

A file you put in your project folder that gives Claude persistent instructions it reads at the start of every session. Like a sticky note for your AI assistant.

Command

A shortcut you type starting with / (e.g., /weather-orchestrator) to trigger a set of pre-written instructions. Like speed dial.

Context Window

The amount of conversation Claude can "remember" at once during a session. Think of it as short-term memory — it has a limit, and older parts get compressed when it fills up.

Frontmatter

Settings written at the very top of a file, between two --- lines. It's how you configure commands, agents, and skills — like filling out a form header before the main content.

Git Worktree

A separate copy of your code that lets Claude (or you) work on a different version without affecting the main one. Used when multiple agents work in parallel.

Hooks

Small scripts that run automatically when specific events happen — like a doorbell that rings when someone arrives. Example: play a sound when Claude finishes a task.

MCP

Model Context Protocol — a standard way to connect Claude to external tools like web browsers, databases, or APIs. Think of it as plugging in accessories.

Model

The AI brain powering Claude. Different models have different strengths: Haiku is fast, Sonnet is balanced, Opus is the most capable.

Monorepo

A single code repository that contains multiple projects or packages inside it, rather than using separate repositories for each.

Orchestration

Coordinating multiple components (commands, agents, skills) to complete a task together in sequence — like a conductor directing an orchestra.

Permissions

Rules controlling what Claude is allowed to do — edit files, run terminal commands, access the internet, etc. You approve or deny these as Claude works.

Plugin

A downloadable package that adds new skills, agents, hooks, or tools to Claude Code. Like installing an app on your phone.

Progressive Disclosure

Giving Claude only the information it needs for the current task, rather than dumping everything at once. Keeps things focused and efficient.

Rules

Instruction files in .claude/rules/ that Claude follows automatically for specific file types or folders. Like house rules that apply in certain rooms.

Sandboxing

Restricting what Claude can access (files, network, commands) for safety. Like childproofing — it prevents accidental damage.

Settings

Configuration files (.claude/settings.json) that control how Claude Code behaves — permissions, which model to use, what tools are available, and more.

Skill

A bundle of instructions that teaches Claude how to do something specific. Stored in .claude/skills/. Think of it as an instruction manual Claude can pick up and follow.

Sub-Agent

An independent copy of Claude that handles a specific subtask and reports back. Like delegating a job to a coworker — they work separately and return with the result.

Tokens

The units Claude uses to measure text. Roughly 1 token = 3/4 of a word. Used to calculate costs and context window usage.

YAML

A simple text format for configuration, using key: value pairs. Used in frontmatter. Pronounced "YAM-ul."

Tips from Creators

10 practical tips for working effectively with Claude Code

Tip 1

Keep CLAUDE.md under 200 lines. Shorter instructions are followed more reliably. Split large docs into rules files for specific contexts.

Tip 2

Use /compact at ~50% context usage. Don't wait until the context window is full. Proactive compaction preserves important details.

Tip 3

Start with plan mode for complex tasks. Let Claude think through the approach before writing code. Switch to act mode once the plan looks right.

Tip 4

Use human-gated task lists for multi-step work. Ask Claude to propose a task list, review it, then have it execute step by step with your approval.

Tip 5

Break tasks small enough to complete in under 50% context. If a subtask might use more than half the context window, it's too big — split it further.

Tip 6

Use /doctor for diagnostics. When something isn't working, run /doctor first. It checks your configuration, MCP servers, and permissions.

Tip 7

Run long commands as background tasks. Long-running terminal processes are easier to monitor and don't block your conversation with Claude.

Tip 8

Use browser MCPs for inspecting console logs. Tools like Playwright or Chrome DevTools MCP let Claude see what's happening in the browser directly.

Tip 9

Provide screenshots when reporting visual issues. Claude can analyze images. A screenshot is worth a thousand words of description.

Tip 10

Create feature-specific agents rather than general ones. Focused agents with preloaded skills outperform catch-all agents. Progressive disclosure keeps each one efficient.
